Cross Site Request Forgery Attack

Cross-Site Request Forgery (CSRF) is a web security vulnerability that occurs when a malicious website, email, blog, instant message, or program causes a user’s web browser to perform an unwanted action on a trusted site when the user is authenticated.

How a CSRF Attack Works:

  1. Victim logs in: The victim logs into a trusted website, such as an online banking portal or an e-commerce site.
  2. Session token: The website assigns the victim a session token, which is used to identify the victim and maintain their session.
  3. Malicious request: An attacker crafts a malicious request that contains the victim’s session token. This request is disguised as a legitimate request from the trusted website.
  4. Victim’s browser: The victim unintentionally clicks on a link or opens an attachment that triggers the malicious request. Their browser sends the forged request to the trusted website.
  5. Forged request executed: The trusted website receives the forged request and executes it, believing it to be a legitimate request from the victim.

Impact of CSRF:

  • Unauthorized transactions: Attackers can use CSRF to perform unauthorized actions on a victim’s behalf, such as transferring funds, making purchases, or changing account settings.
  • Data theft: CSRF can be used to steal sensitive information, such as credit card details or personal data.
  • Account compromise: CSRF can compromise a victim’s account, allowing the attacker to gain access to their personal information and resources.

Prevention:

  • Synchronizer Token Pattern (STP): This is a common technique to prevent CSRF by adding a unique, unpredictable, secret, session-specific token to every web request.
  • Double Submit Cookie (DSC): This technique involves setting a random value in both a cookie and a hidden field in the HTML form. The server then checks that the values match before processing the request.
  • Checking the HTTP Referer header: While not entirely reliable, as it can be modified or absent, checking the Referer header can provide some level of protection against CSRF.
  • Educating users: Educate users about the risks of CSRF and encourage them to be cautious of clicking on suspicious links or opening attachments from unknown sources.
